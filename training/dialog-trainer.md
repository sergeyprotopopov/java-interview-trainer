## 🟥 Tier S

**В: Какие есть уровни изоляции транзакций и какой дефолтный?**
О: Четыре по стандарту SQL: READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE. Чем выше — тем меньше аномалий, но ниже параллелизм. Аномалии: грязное чтение, неповторяемое чтение, фантомы. Дефолт: в PostgreSQL и Oracle — READ COMMITTED, в MySQL (InnoDB) — REPEATABLE READ.

**В: Что такое грязное чтение и почему в PostgreSQL READ UNCOMMITTED ведёт себя как READ COMMITTED? ‼️**
О: Грязное чтение — это когда транзакция видит незакоммиченные изменения другой. В PostgreSQL благодаря MVCC каждая транзакция видит снимок данных, и незакоммиченные версии ей просто не видны. Поэтому READ UNCOMMITTED фактически работает как READ COMMITTED — настоящего dirty read нет.

**В: Делает ли `volatile` операции атомарными? ‼️**
О: Нет. `volatile` гарантирует только видимость (чтение/запись напрямую из main memory) и запрет переупорядочивания. Но `count++` — это read-modify-write, три операции, и они не атомарны. Для атомарности нужен `AtomicInteger`, `synchronized` или `Lock`.

**В: Что такое happens-before?**
О: Это гарантия модели памяти Java: если действие A happens-before B, то результаты A видны в B. Примеры: разблокировка монитора happens-before его захвата, запись в `volatile` happens-before последующего чтения этого поля, старт потока happens-before его кода. Это не про «порядок во времени», а про гарантию видимости и упорядочивания.

**В: Как устроен Spring DI-контейнер внутри и как решает циклические зависимости?**
О: При старте контекст сканирует пакеты, находит бины (по аннотациям `@Component`/`@Service` и `@Configuration`/`@Bean`), строит `BeanDefinition`, затем создаёт бины и внедряет зависимости (через конструктор/сеттер/поле). Циклические зависимости через конструктор он разрешить не может — кидает ошибку; через сеттер/поле разрешает за счёт раннего раскрытия ссылки (early reference) или можно пометить `@Lazy`, чтобы внедрить прокси.

**В: Что такое deadlock и как избежать при переводах между счетами?**
О: Deadlock — когда два потока держат по ресурсу и ждут ресурс друг друга. В переводах классика: поток 1 лочит счёт A потом B, поток 2 — B потом A. Решение: всегда захватывать локи в одном глобальном порядке (например по id счёта), использовать `tryLock` с таймаутом, или стратегии Wait-Die/Wound-Wait.

**В: Оптимистичная vs пессимистичная блокировка в БД, что такое SELECT FOR UPDATE?**
О: Пессимистичная — блокируем строку сразу (`SELECT ... FOR UPDATE`), другие ждут; хорошо при высокой конкуренции. Оптимистичная — не блокируем, а проверяем версию (`version`/timestamp) при коммите, и если кто-то изменил — откатываем и повторяем; хорошо при редких конфликтах. `SELECT FOR UPDATE` берёт строковую блокировку до конца транзакции.

## 🟧 Tier A

**В: Каков контракт equals/hashCode?**
О: Если `a.equals(b)` true, то `a.hashCode() == b.hashCode()` обязан быть равен. Обратное не обязательно (коллизии допустимы). Если переопределяешь `equals`, обязан переопределить `hashCode`, иначе объект «потеряется» в HashMap/HashSet. equals должен быть рефлексивным, симметричным, транзитивным.

**В: Почему ключ в HashMap должен быть неизменяемым?**
О: Хэш считается при добавлении и определяет бакет. Если после вставки поле, влияющее на `hashCode`, изменится, объект окажется «не в том» бакете, и `get` его не найдёт. Поэтому ключи делают immutable (String, Integer, Long).

**В: Назови принципы ООП.**
О: Инкапсуляция (скрытие состояния за методами), наследование (переиспользование), полиморфизм (один интерфейс — разные реализации), абстракция (выделение существенного). На практике предпочитают композицию наследованию.

**В: Композиция vs агрегация vs наследование?**
О: Наследование — «is-a», жёсткая связь. Композиция — «has-a» с владением, часть не живёт без целого (двигатель внутри машины). Агрегация — «has-a» без владения, части живут отдельно (студенты в группе). Предпочитают композицию: гибче и слабее связанность.

**В: Интерфейс vs абстрактный класс?**
О: Абстрактный класс — частичная реализация + состояние (поля), одиночное наследование. Интерфейс — контракт, множественная реализация, с Java 8 есть `default`/`static` методы, но без состояния. Интерфейс — когда нужен полиморфизм без общей базы; абстрактный класс — когда есть общий код и состояние.

**В: ArrayList vs LinkedList?**
О: ArrayList — массив, доступ по индексу O(1), вставка/удаление в середину O(n), кэш-френдли — почти всегда быстрее на практике. LinkedList — двусвязный список, O(1) вставка по уже найденному узлу, но O(n) доступ по индексу и большой оверхед на узлы. Реально LinkedList нужен редко.

**В: Как сделать LRU-кэш?**
О: Через `LinkedHashMap` с `accessOrder=true` и переопределением `removeEldestEntry` — он сам выселяет самый старый элемент при превышении размера. Внутри это хэш-таблица + двусвязный список порядка доступа.

**В: Comparable vs Comparator?**
О: `Comparable` — естественный порядок, метод `compareTo`, реализуется в самом классе. `Comparator` — внешний порядок, метод `compare`, можно несколько разных, удобно лямбдами: `Comparator.comparing(User::getAge).thenComparing(User::getName)`.

**В: Какие есть сборщики мусора?**
О: Serial (однопоточный), Parallel (пропускной, многопоточный), G1 (дефолт, регионы, предсказуемые паузы), ZGC и Shenandoah (низколатентные, паузы <1мс, подходят для больших куч). Общий принцип: найти недостижимые объекты (от GC roots) и освободить, обычно поколенчески (young/old).

**В: Размеры и ограничения числовых типов?**
О: `int` — 32 бита, ±2.1 млрд; `long` — 64 бита; `float` — 32, `double` — 64 бита (неточные, нельзя для денег); `BigDecimal` — произвольная точность, для денег. Переполнение `int` молча заворачивается — для счётчиков/денег брать `long`/`BigDecimal`.

**В: Какие бывают индексы и какова цена их наличия?**
О: B-Tree (дефолт, диапазоны и сортировка), Hash (только равенство), составной (по нескольким колонкам, важен порядок и left-prefix), частичный (с условием), уникальный, полнотекстовый, кластеризованный (физический порядок строк). Цена: ускоряют чтение, но замедляют запись (INSERT/UPDATE) и занимают место.

**В: Что такое Reflection и цепочка ClassLoader'ов?**
О: Reflection — доступ к структуре класса в рантайме (поля, методы, аннотации с `RetentionPolicy.RUNTIME`), создание объектов и вызов методов динамически. ClassLoader'ы образуют иерархию с делегированием родителю: Bootstrap → Platform → Application. Сначала спрашивают родителя, потом грузят сами.

**В: Что такое MVCC в PostgreSQL?**
О: Multi-Version Concurrency Control — на каждое изменение создаётся новая версия строки, а старые видны транзакциям с их снимком. Читатели не блокируют писателей и наоборот. SERIALIZABLE в PG реализован как Serializable Snapshot Isolation — отслеживает опасные пересечения и откатывает конфликтующую транзакцию.

**В: Как обеспечивается атомарность Atomic/CAS? ‼️**
О: Через CAS — Compare-And-Swap. Это операция «сравни текущее значение с ожидаемым и, если совпало, замени, иначе повтори». Опирается на инструкцию процессора (CMPXCHG) и в JVM реализована через `Unsafe`/`VarHandle`. Это lock-free: без блокировок, в цикле повторов. Не «просто метод» и не обычный `value++`.

**В: Назови три типа паттернов проектирования с примерами.**
О: Порождающие (создание объектов): Singleton, Factory Method, Builder. Структурные (композиция): Adapter, Decorator, Facade, Proxy. Поведенческие (взаимодействие): Strategy, Observer, Command, Template Method.

**В: OSI: кто что делает — DNS, ARP, NAT, DHCP? ‼️**
О: DNS — имя в IP. ARP — IP в MAC внутри локальной сети. DHCP — автоматически выдаёт устройству сетевые настройки (IP, маска, шлюз). NAT — подменяет приватный IP на публичный на выходе из сети. Маршрутизация между сетями — сетевой уровень (L3), а MAC и кадры в локалке — канальный (L2).

## 🟨 Tier B

**В: Как устроена Kafka и чем отличается от обычной очереди (JMS/RabbitMQ)?**
О: Kafka — распределённый лог. Сообщения в топиках, разбитых на партиции, хранятся на диске и не удаляются после чтения (retention). Потребители читают по своему offset, есть consumer groups для масштабирования. Отличие от JMS/MQ: не «доставил и удалил», а durable-лог с перечитыванием, упорядоченность в рамках партиции, высокая пропускная способность.

**В: Как сделать REST надёжным?**
О: Идемпотентность (повтор не меняет результат — особенно для POST через idempotency key), таймауты на вызовы, ретраи с backoff, circuit breaker для защиты от каскадных отказов, корректные статус-коды и DLQ для асинхронщины.

**В: System design: лента новостей — pull или push?**
О: Pull (fan-out on read) — лента собирается при запросе из постов тех, на кого подписан; дёшево по записи, дорого по чтению. Push (fan-out on write) — пост сразу разлетается в ленты подписчиков; быстрое чтение, но дорого для «звёзд» с миллионами фолловеров. На практике гибрид: push для обычных, pull для крупных аккаунтов, плюс кэш и ранжирование.

**В: Что такое Saga?**
О: Паттерн для распределённых транзакций без 2PC: бизнес-операция разбивается на локальные транзакции в каждом сервисе, и на каждый шаг есть компенсирующее действие. Если шаг падает — запускаются компенсации для отката предыдущих. Бывает оркестрация (центральный координатор) и хореография (через события).

**В: Что и как мокать в тестах (Mockito)?**
О: Мокаем внешние зависимости тестируемого класса — репозитории, клиенты других сервисов, чтобы изолировать логику. `when(...).thenReturn(...)` задаёт поведение, `verify(...)` проверяет вызовы. Не мокаем то, что тестируем, и не мокаем простые value-объекты.

**В: Зачем нужна реактивность (WebFlux/Reactor)? ‼️**
О: Не «для UI». Это про неблокирующий I/O: малым числом потоков обрабатывать много одновременных соединений (high concurrency, стриминг). `Mono`/`Flux` — асинхронные потоки 0..1 / 0..N с backpressure. Выигрыш именно при I/O-bound нагрузке и интеграциях, не на CPU-задачах.

**В: ConcurrentHashMap vs synchronizedMap, что такое ThreadLocal?**
О: `synchronizedMap` лочит всю карту на каждую операцию. `ConcurrentHashMap` лочит по сегментам/бинам — гораздо лучше масштабируется, чтения почти без блокировок. `ThreadLocal` — переменная, у каждого потока своя копия; удобно для контекста, но в пулах потоков надо чистить, иначе утечка.

**В: Stream API: промежуточные vs терминальные операции, ленивость?**
О: Промежуточные (`filter`, `map`, `sorted`) возвращают стрим и ленивы — не выполняются, пока нет терминальной. Терминальные (`collect`, `forEach`, `reduce`, `count`) запускают конвейер. Ленивость даёт оптимизации: short-circuit (`findFirst`, `limit`), один проход по данным.

**В: Что такое DDD и bounded context?**
О: DDD — проектирование от предметной области и единого языка с экспертами. Bounded context — граница, внутри которой термины модели однозначны; в разных контекстах один и тот же «Заказ» может значить разное. Это не просто пакет — это семантическая граница со своей моделью и языком.

**В: CQRS — кто меняет состояние? ‼️**
О: **Command** меняет состояние (пишет), без возврата данных. **Query** только читает, без побочных эффектов. Разделяем модели чтения и записи; часто разные хранилища и масштабирование независимо.

**В: Репликация vs шардирование vs масштабирование?**
О: Репликация — копии одних данных (надёжность + чтение с реплик). Шардирование — данные разбиты по разным узлам по ключу (горизонтальное масштабирование объёма/нагрузки на запись). Вертикальное масштабирование — мощнее одна машина; горизонтальное — больше машин.

**В: Как написать потокобезопасный Singleton?**
О: Простой — через статическое поле/enum (потокобезопасно за счёт загрузки класса). Ленивый — double-checked locking: поле `volatile`, проверка null, `synchronized`-блок, повторная проверка. `volatile` тут обязателен, чтобы не отдать недоинициализированный объект.

**В: SQL: разница WHERE и HAVING, зачем GROUP BY?**
О: `GROUP BY` группирует строки для агрегатов (`COUNT`, `SUM`). `WHERE` фильтрует строки до группировки, `HAVING` — после, по агрегатам (например `HAVING COUNT(*) > 5`).

## 🟩 Tier C

**В: String, StringBuilder, String Pool?**
О: String неизменяем (immutable) — потокобезопасен, кэшируется. String Pool хранит литералы, одинаковые строки переиспользуются (`intern`). Для конкатенации в цикле — `StringBuilder` (не потокобезопасный, быстрый) или `StringBuffer` (синхронизированный).

**В: HTTP/2 vs HTTP/1.1, что такое gRPC?**
О: HTTP/2 — мультиплексирование запросов в одном соединении, бинарные фреймы, сжатие заголовков, server push — нет head-of-line блокировки на уровне HTTP. gRPC — RPC поверх HTTP/2 с Protobuf: компактно, быстро, со стримингом и кодогенерацией контрактов.

**В: Контейнеры vs виртуализация?**
О: VM виртуализирует железо и тащит свою ОС — тяжело, изолированно. Контейнеры делят ядро хоста, изоляция через namespaces (видимость) и cgroups (ресурсы) — легче и быстрее стартуют. Docker — образы/контейнеры, Kubernetes — оркестрация (поды, масштабирование, самовосстановление).

**В: Исключения: checked vs unchecked, final/finally/finalize?**
О: Checked (наследники Exception) — компилятор требует обработать/объявить. Unchecked (RuntimeException) — ошибки логики. Error — фатальные (OOM, StackOverflow). `final` — нельзя менять/наследовать/переопределять; `finally` — блок, выполняется всегда; `finalize` — устаревший метод перед GC, не использовать.

**В: Как избавиться от null / зачем Optional?**
О: `Optional` — явный контейнер «значение может отсутствовать», заставляет обработать пустоту (`map`, `orElse`, `orElseThrow`). Использовать для возвращаемых значений, не для полей/параметров. Альтернативы: проверки на null, аннотации `@Nullable`/`@NonNull`, паттерн Null Object.

**В: Утечка памяти: heap или stack? ‼️**
О: Утечка — про heap: недостижимые с точки зрения логики объекты держатся живыми ссылками (статические коллекции, незакрытые ресурсы, ThreadLocal в пуле) и GC их не собирает. В stack утечки нет — там переполнение `StackOverflowError` при слишком глубокой рекурсии.

**В: Синхронная vs асинхронная репликация — что с latency?**
О: Синхронная — мастер ждёт подтверждения реплики перед коммитом: сильнее консистентность, но выше latency и риск простоя при медленной реплике. Асинхронная — мастер коммитит сразу: быстро, но возможна потеря последних транзакций при сбое и отставание реплик.

**В: Партиционирование vs шардирование?**
О: Партиционирование — разбиение таблицы на части в пределах одной БД/инстанса (по диапазону/хэшу/списку). Шардирование — распределение данных по разным узлам/серверам. Шардинг — это, по сути, партиционирование через сеть для горизонтального масштабирования.

**В: Почему `Arrays.asList().add()` бросает исключение?**
О: `Arrays.asList` возвращает фиксированный список-обёртку над массивом; размер менять нельзя, поэтому `add`/`remove` кидают `UnsupportedOperationException`. Чтобы изменять — оберни: `new ArrayList<>(Arrays.asList(...))`.
