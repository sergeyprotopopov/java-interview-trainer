# ⚠ Слабые места (что подтянуть)

На реальном собеседовании были запинки/ошибки в этих темах. Прогони их в первую очередь.

## 1. CQRS — кто меняет состояние

❌ Была ошибка: «Query меняет состояние».
✅ Правильно: состояние меняет **Command**. **Query** только читает, без побочных эффектов.
→ [05-ddd-cqrs-reactive.md](topics/05-ddd-cqrs-reactive.md)

## 2. Сетевые протоколы — путаница ARP / DNS / NAT / DHCP

Запомни шпаргалку:
- **DNS**: имя → IP
- **ARP**: IP → MAC (внутри локальной сети)
- **DHCP**: автоматическая выдача сетевых настроек устройству
- **NAT**: внутренний приватный IP → внешний публичный IP

И уровни: **маршрутизация — это сетевой уровень**, а не канальный (канальный — MAC/кадры в локалке).
→ [07-containers-networking.md](topics/07-containers-networking.md)

## 3. Реализация Atomic / CAS

Интервьюер несколько раз тянул к сути реализации, не к применению.
✅ Атомарность обеспечивается **CAS на уровне JVM (`Unsafe`/`VarHandle`) и процессора**, а не «просто методом» или обычным `value++`.
✅ CAS = прочитать → заменить только если не изменилось → иначе повторить (lock-free).
→ [10-concurrency.md](topics/10-concurrency.md)

## 4. Грязное чтение в Postgres

✅ В Postgres `READ UNCOMMITTED` фактически работает как `READ COMMITTED` (из-за MVCC) — реального dirty read нет.
→ [02-transactions-isolation.md](topics/02-transactions-isolation.md)

## 5. Утечки памяти: heap vs stack

✅ Утечка памяти — это про **heap** (недостижимые объекты держатся ссылками).
✅ В **stack** не утечка, а переполнение (`StackOverflowError` при глубокой рекурсии).
→ [08-jvm-memory.md](topics/08-jvm-memory.md)

## 6. volatile не делает операции атомарными

✅ `volatile` даёт видимость, но `count++` (read-modify-write) всё равно небезопасен → нужен `Atomic`/`synchronized`/`Lock`.
→ [10-concurrency.md](topics/10-concurrency.md)
