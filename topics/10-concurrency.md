# 10. Многопоточность и синхронизация

## Основы

**В: Поток vs процесс?**
О: Процесс — отдельная программа со своей памятью и ресурсами. Поток — единица выполнения внутри процесса, память у потоков общая. Между процессами обмен дороже (IPC), между потоками — через общую память + синхронизация.

**В: Как процессы взаимодействуют?**
О: Через IPC: сокеты, файлы, pipe/named pipe, shared memory, очереди сообщений, БД, HTTP/RPC. У процессов память раздельная.

**В: Как потоки взаимодействуют?**
О: Через общую память: общие объекты, `volatile`, `synchronized`, `Lock`, `Atomic*`, `BlockingQueue`, и сигналы (`wait/notify`, `CountDownLatch`, `Semaphore`).

**В: Виды многозадачности?**
О: Кооперативная (поток сам уступает), вытесняющая (планировщик ОС переключает по кванту/приоритету), параллельная (реально одновременно на разных ядрах), конкурентная (вперемешку).

**В: Кто отвечает за scheduling потоков?**
О: Операционная система. JVM создаёт поток (`Thread`), а планировщик ОС раздаёт CPU по квантам/приоритетам.

**В: Что такое Thread?**
О: Единица выполнения внутри процесса. `start()` создаёт новый поток (не `run()`). Потоки делят память процесса → нужна синхронизация.

**В: Что такое виртуальные потоки?**
О: Лёгкие потоки под управлением JVM, не привязаны 1:1 к OS-потокам. При блокировке на I/O освобождают carrier thread. Польза для I/O-нагрузки, не для CPU-bound.

## Синхронизация

**В: Зачем нужен synchronized?**
О: Взаимное исключение (один поток в критической секции), видимость изменений, упорядочивание. Нужен при общем изменяемом состоянии. Не нужен для immutable или при `Atomic`/concurrent-коллекциях.

**В: wait/notify — как работают?**
О: Только внутри `synchronized` (иначе `IllegalMonitorStateException`), работают через монитор объекта. `wait()` освобождает монитор и ждёт. `notify()` будит один поток, `notifyAll()` — все, но продолжат только после повторного захвата монитора. Использовать `wait()` в цикле `while`, а не `if`.

**В: volatile — что даёт?**
О: Видимость изменений между потоками и запрет опасного переупорядочивания. ⚠ НЕ делает составные операции атомарными (`count++` всё равно небезопасен). Для флагов: один пишет — другие читают.

## Atomic, CAS, Lock

**В: Atomic — зачем и в чём преимущество?**
О: Thread-safe доступ к одному значению без блокировки, через **CAS** (lock-free). Преимущество: нет блокировки/ожидания потока, быстрее `synchronized` для простых операций (счётчики, флаги, `compareAndSet`). Ограничение: только одна переменная, не для сложной логики на несколько полей.

**В: Что такое CAS?**
О: ⚠ Compare-And-Swap: прочитать значение → заменить на новое только если оно не изменилось → иначе повторить попытку. Lock-free, реализован на уровне JVM/CPU-инструкции. При высокой конкуренции возможны много ретраев.

**В: Где «лежит» реализация Atomic?**
О: API — `AtomicInteger`/`AtomicReference`; атомарность — на уровне JVM (`Unsafe`/`VarHandle`) и процессора (инструкция CAS). Не в обычном Java-коде `value++`.

**В: Atomic vs Lock/mutex?**
О: Atomic — простые операции над одним значением без блокировки. Lock — защита целой критической секции, гибче (tryLock, таймаут, Condition), для сложной логики на несколько шагов. Если просто `count++` → Atomic; если критическая секция с несколькими действиями → Lock/synchronized.

## Проблемы

**В: Что такое deadlock?**
О: Взаимная блокировка: поток держит ресурс A и ждёт B, другой держит B и ждёт A — никто не продолжает. Избегают: захват ресурсов в одном порядке, таймауты, меньше локов.

**В: Что такое race condition?**
О: Результат зависит от порядка выполнения потоков из-за одновременного доступа к общим данным без синхронизации. Пример: два потока читают одно значение и оба инкрементят — один инкремент теряется. Лечится `synchronized`/`Lock`/`Atomic`.

**В: Практический пример параллелизма?**
О: Несколько долгих сервисов/AI-моделей дёргаются параллельно через `CompletableFuture`/пул потоков, потом результаты агрегируются. Общее время ≈ самый медленный вызов, а не сумма.
