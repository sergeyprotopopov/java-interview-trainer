# 08. JVM: модель памяти, GC, утечки

**В: Как устроена модель памяти JVM?**
О: Heap (объекты, общий для потоков, чистит GC), Stack (у каждого потока свой: кадры вызовов, локальные переменные, ссылки), Metaspace (метаданные классов), плюс PC Register и Native Method Stack.

**В: Heap vs Stack — что где?**
О: Объекты (`new`) — в heap. Локальные переменные и ссылки на объекты — в stack. Переполнение heap → `OutOfMemoryError`; слишком глубокая рекурсия → `StackOverflowError`.

**В: Stack — для процессов или потоков?**
О: Для потоков. У каждого потока свой stack.

**В: Можно ли выделять память вне heap и stack?**
О: Да — native memory: `DirectByteBuffer` (NIO/сетевые буферы), JNI/нативные библиотеки, Metaspace. ⚠ Эта память не управляется GC, при злоупотреблении — утечки вне heap.

**В: Что такое Java Memory Model (JMM) и happens-before?**
О: JMM описывает видимость, атомарность и порядок операций между потоками. Главное правило — **happens-before**: если оно установлено, запись одного потока гарантированно видна другому. Примеры: выход из `synchronized` → вход в тот же; запись `volatile` → чтение этого `volatile`; `start()`/`join()` потока.

**В: Бывают ли утечки памяти в Java?**
О: Да. Объекты логически не нужны, но на них есть достижимые ссылки, и GC их не освобождает. Частые причины: статические коллекции/кэши без лимита, неотписанные listeners/callbacks, `ThreadLocal` без очистки, долгоживущие ссылки из синглтонов. ⚠ Утечки — в heap; в stack — не утечка, а переполнение.

**В: Когда работает GC?**
О: Запускается, когда выгодно JVM по памяти/нагрузке. Удаляет только недостижимые объекты. Память освобождается не мгновенно — это нормальная работа GC, а не утечка.

**В: Что такое ClassLoader и зачем нужен?**
О: Загружает классы в JVM в рантайме (ленивая загрузка), читает байткод. Цепочка: Bootstrap → Platform → Application → Custom. Сначала ищет у родителя (delegation). Нужен для изоляции, безопасности, плагинов, динамической загрузки.
