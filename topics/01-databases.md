# 01. Базы данных: Postgres vs Mongo

**В: Чем Mongo лучше Postgres в плане масштабирования?**
О: У Mongo шардирование (горизонтальное масштабирование) встроено из коробки. В Postgres это тоже возможно, но сложнее: через партиционирование, Citus или логику на уровне приложения.

**В: Куда хранить деньги/счета/транзакции — Postgres или Mongo?**
О: Postgres. Сильная ACID-модель, транзакции, ограничения целостности, уровни изоляции. Для финансов нужна строгая консистентность.

**В: Обеспечивает ли JSON консистентность?**
О: Нет. JSON — просто формат. Целостность дают БД, транзакции и приложение. Для критичных данных: основные поля — в колонках, JSON — только для редких/расширяемых атрибутов.

**В: Когда выбирать Postgres, когда Mongo?**
О:
- **Postgres** — структурированные данные, связи, JOIN, транзакции, сложные запросы, отчёты, критичность данных.
- **Mongo** — гибкая/часто меняющаяся схема, документы целиком, ранняя стадия продукта, логи/события/каталоги/профили.

**В: Что такое неструктурированные данные и где их хранить?**
О: Данные с заранее неизвестной/часто меняющейся структурой. Хранят в NoSQL, документоориентированных БД, объектном хранилище (S3) или как JSON/BLOB в реляционной БД. Минусы: сложнее валидировать и делать сложные запросы.

**В: Работал ли с NoSQL?**
О: Да, с MongoDB (~полгода): CRUD, индексы, агрегации, базовая оптимизация. Понимаю: нет привычных JOIN, схему нужно продумывать сразу, при неправильной модели быстро упираешься в производительность.
